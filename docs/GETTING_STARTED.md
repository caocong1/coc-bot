# 快速开始指南

## 环境要求

- [Bun](https://bun.sh) >= 1.1
- NapCatQQ（已启动，配置 WebSocket + HTTP）
- 阿里云百炼 API Key（DashScope）

---

## 1. 安装依赖

```bash
bun install
```

---

## 2. 配置环境变量

```bash
cp .env.example .env
```

编辑 `.env`：

```env
DASHSCOPE_API_KEY=sk-xxxx        # 阿里云百炼 API Key（AI KP 必须）
NAPCAT_WS_URL=ws://127.0.0.1:3003
NAPCAT_HTTP_URL=http://127.0.0.1:3002
NAPCAT_TOKEN=                    # NapCat 鉴权 Token（可留空）
SERVER_PORT=28765                # HTTP 管理接口端口
DATABASE_PATH=./data/storage/coc-bot.db  # SQLite 路径（可选）
```

---

## 3. 导入知识库（可选，AI KP 需要）

将 CoC7 规则书 PDF、模组 PDF 放到项目根目录，然后：

```bash
# 提取文字并切片
bun run import-pdfs

# 构建向量索引（推荐加 --embed 使用真实语义向量）
bun run build-indexes:rules    # 规则书索引
bun run build-indexes:scenario # 模组索引

# 或一键全部构建
bun run build-indexes:all
```

> **注意**：`--embed` 需要 `DASHSCOPE_API_KEY`。不加则使用哈希伪向量（检索精度低，适合测试）。

---

## 4. 启动 Bot

```bash
bun run dev
```

成功后控制台会显示：

```
=== CoC Bot 启动 ===
[Bot] logged in as <昵称> (<QQ号>)
[Server] HTTP on http://localhost:28765
=== CoC Bot 就绪 ===
```

---

## 5. Web 控制台

Bot 内置 Web 控制台，分为**玩家端**和**管理端**，通过浏览器访问 `http://localhost:28765`。

### 玩家端登录

**私聊机器人**发：

```
.web login
```

机器人回复一个带 24h 有效 token 的直达链接，点击即可登录玩家端。

> ⚠️ `.web login` 只在私聊有效，群内发送会提示去私聊。

### 玩家端功能

| 页面 | 功能 |
|------|------|
| 我的角色卡 | 查看/新建/编辑角色卡（两步车卡向导） |
| 可用模组 | 浏览已导入模组（简介/时代/职业约束/主属性总点要求） |
| 房间 | 创建/加入房间，选 PC，从 Web 触发开团；房间详情支持在“房间信息 / 消息历史”标签间切换，查看完整跑团记录 |

### Web 车卡流程（推荐）

1. 进入「我的角色卡」→「新建角色卡」
2. **Step 1**：点「🎲 掷骰（4组）」生成 4 组属性对比表，选一组
3. **Step 2**：填姓名/职业/时代，分配职业技能点和兴趣点，填写背景故事
4. 点「💾 保存」→ 自动同步到 bot，无需发任何 QQ 命令

### 管理端登录

在 `.env` 中设置 `ADMIN_SECRET`，然后在管理端登录页输入密码。

### 管理端功能

| 页面 | 功能 |
|------|------|
| 会话监控 | 查看所有群的跑团状态，实时消息流 |
| 跑团管理 | start/pause/resume/stop，手动切换场景，线索管理 |
| KP Studio | KP 人格模板管理——五维滑块调节（基调/灵活度/引导度/致命度/节奏）+ 自定义设定语 |
| 模组管理 | 创建模组，上传文档/图片，AI 生成图片 |
| 房间管理 | 查看所有房间、消息历史、运行态、KP 设定和强制删除 |
| 知识库 | 上传规则书 PDF，查看导入状态 |

## 7. AI KP 跑团模式

### 典型流程

```
# ── 准备阶段（在 QQ 群） ──────────────────────────────

# 查看可用模组（含模组 ID）
.mod list

# KP 创建跑团房间（可选绑定模组 ID）
.room create 与苏珊共进晚餐 abc12345

# 玩家加入房间，机器人私发 Web 车卡链接
.room join f3a9c1b2

# ── 车卡阶段（在 Web 控制台） ────────────────────────
# 1. 私聊机器人发 .web login 获取登录链接
# 2. 打开链接 → 「我的角色卡」→「新建角色卡」掷骰车卡
# 3. 进入「房间」→ 选择本次跑团使用的角色卡
# 4. 房间创建者可在房间详情里调整主属性总点要求；默认继承模组值 460，也可留空

# ── 开团（在 QQ 群） ─────────────────────────────────

# KP 在目标群内开团
.room start f3a9c1b2

# 正常跑团... 玩家发言，AI KP 自动判断是否接话
# 需要投骰时玩家自行 .ra 或 .r，KP AI 自动接话

# 中途需要暂停（保存所有进度）
.room pause

# 下次继续（AI 生成回顾摘要）
.room resume

# 彻底结束
.room stop
```

### 房间命令

| 命令 | 说明 |
|------|------|
| `.mod list` | 列出可用模组（每页 5 个） |
| `.mod info <ID>` | 查看模组详情 |
| `.room create <名称> [模组ID]` | 创建房间 |
| `.room join <房间ID>` | 加入房间，私发 Web 车卡链接 |
| `.room pc <角色卡名或ID>` | 在目标群里为当前房间绑定自己的 PC |
| `.room relation list` | 查看当前房间的人物关系 |
| `.room relation set <QQ号> <关系类型> [备注]` | 设置你与某位队友的显式关系 |
| `.room relation clear <QQ号>` | 清除你与某位队友的显式关系 |
| `.room start <房间ID>` | 在当前群开始跑团（仅创建者） |
| `.room pause` | 暂停跑团，保存所有对话/线索/场景进度 |
| `.room resume` | 继续最近暂停的跑团，AI 生成回顾摘要 |
| `.room stop` | 彻底结束跑团 |
| `.room list` | 查看我参与的活跃房间 |
| `.room info <房间ID>` | 查看房间成员和状态 |

### 场景频道命令

用于手动分线、秘密团或需要显式调度频道的高级场景：

| 命令 | 说明 |
|------|------|
| `.scene list` | 查看当前焦点频道、活跃频道、频道内调查员、待结算检定 |
| `.scene focus <频道名>` | 将处理焦点切到指定频道，并立刻处理该频道积压消息 |
| `.scene join <频道名>` | 把自己切换到指定频道 |
| `.scene move <QQ号> <频道名>` | 将指定玩家移动到某个频道 |
| `.scene merge <来源频道> [目标频道]` | 合并频道，默认并回 `main` |
| `.scene clear [频道名]` | 清除频道的紧急标记 |

> 普通公开团默认统一在群里公开推进，不会自动创建分线频道。只有手动 `.scene` 分线或秘密团时，频道机制才会真正进入主流程。
>
> 当非焦点频道出现战斗、追逐、理智冲击、陷阱等高危情况时，bot 会在群里提示切换 `focus`。

### 开场导演与关系预设

- 房间支持预设**人物关系**，AI KP 开场会优先参考这些信息。
- 开场导演使用**系统内部全局策略**：默认允许分头开场、自动根据职业/背景/人物关系判断同场或分头进场、允许高强度模组内扩写，私密引子维持轻量。
- 没有显式关系时，开场导演会根据职业、人设和模组入口做轻量推断，但不会回写数据库。
- 默认所有团按**公开团**处理：即使某位调查员暂时单独行动，也仍然在群里公开叙事，其他玩家作为桌边旁观者听见 KP 的描述。
- 只有模组规则包明确标记为**秘密团**时，AI 才会启用私聊推进、秘密分线和更严格的个人秘密处理。
- 开场现在允许：
  - 不同调查员在不同地点、不同时间分批入场
  - 适度补充模组内合理的连接场景、路人和个人化引子
  - 最终给出自然汇合目标，而不是强行把所有人瞬间拉到一起
- 导演策略不会作为房间配置暴露给用户；如果后续需要调整，应视为系统级行为策略，而不是某个房间的手动参数。

### AI KP 介入逻辑

| 触发条件 | KP 行为 |
|----------|---------|
| 骰子结果出现 | 立即基于结果叙述后果 |
| `(...)` / `OOC:` 开头 | 跳出叙事直接答疑 |
| `KP,` / `守秘人,` 开头 | 直接回应 |
| 玩家描述行动（「我想/我要/我试...」） | 判断是否需要检定，告知技能名 |
| 玩家对 NPC 说话 | 以 NPC 身份回应 |
| 玩家互 RP 超过 5 条 | 插入一句氛围描写 |
| 纯玩家互 RP | 静默观察 |

## 8. 模组管理

通过管理端 Web UI 进行，无需命令行：

1. 管理端登录 → 进入「模组管理」
2. 点「＋ 新建模组」，填写名称/简介/时代/职业限制/主属性总点要求（默认 460，可留空）
3. 展开模组卡片 → 「文档」区上传 PDF/TXT，后台自动建索引
4. 「图片」区上传地图/道具图，或点「🤖 AI 生成图片」输入描述生成
5. 玩家端「可用模组」页即可看到，创建房间时可选择绑定

## 9. 数据存储

所有数据存储在 `data/storage/coc-bot.db`（SQLite）：

- `characters` — 角色卡
- `active_cards` — 激活绑定（userId:groupId → characterId）
- `kp_sessions` — 跑团会话（status: running / paused / ended）
- `kp_events` — 跑团事件流（canonical runtime history）
- `kp_scenes` — 当前场景
- `kp_clues` — 线索
- `kp_messages` — 对话历史（含摘要压缩标记）
- `kp_summaries` — 历史摘要
- `kp_pending_rolls` — 等待玩家投骰
- `kp_templates` — 自定义 KP 人格模板（五维参数 + 自定义设定语）
- `scenario_modules` — 模组（名称/简介/时代/职业限制/主属性总点要求）
- `scenario_module_files` — 模组文档和图片（含导入状态）
- `campaign_rooms` — 跑团房间（含 module_id / kp_template_id / kp_custom_prompts）
- `campaign_room_members` — 房间成员与 PC 绑定
- `player_tokens` — 玩家 Web 登录 token
- `user_settings` — 用户设置（默认骰面数/昵称）

知识库存储在 `data/knowledge/`：

```
data/knowledge/
├── manifest.json                      # 文件索引
├── raw/                               # 提取的纯文本
├── chunks/                            # 切片 JSON
├── indexes/                           # 向量索引（.bin）
└── images/modules/{moduleId}/        # 模组图片
```

## 10. 常见问题

### Q: AI KP 没有回复？

检查：
1. `DASHSCOPE_API_KEY` 是否已配置
2. 是否已使用 `.room start` 开团
3. 消息是否触发了介入逻辑（见上表）

### Q: AI KP 思考中继续发消息会怎样？

消息会自动排队。KP 回复当前消息后会把排队的消息合并为一条，再进行一次 AI 回复。不会出现多条"正在思考"或回复交叉。

### Q: AI 调用超时或出错？

`DashScopeClient` 有 60 秒超时保护。AI 生成失败后自动重试一次，仍然失败会返回"⚠️ KP 暂时无法回应，请稍后再试"。

### Q: 群里非房间成员说话会触发 KP 吗？

不会。开团后只有房间成员（`campaign_room_members`）的消息和骰子结果会触发 AI KP。无关联房间的独立 session 不限制。

### Q: 如何加载模组？

通过管理端 Web UI 上传：管理端 → 模组管理 → 新建模组 → 上传 PDF/DOCX。上传后自动建索引，玩家端和 `.mod list` 即可看到。

### Q: 多次跑团如何区分？

每次 `.room start` 创建新 session，数据独立。`.room pause` 后 `.room resume` 会自动恢复最近一次暂停的 session。

### Q: 如何添加新命令？

在 `src/commands` 下创建命令处理器，实现 `CommandHandler` 接口，然后在 `src/server/index.ts` 的 `registry.register(...)` 中注册。
